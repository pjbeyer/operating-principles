## B. The structural recommendation — Integrity as a top-level FLOOR

The research's strongest finding. Today the spec has one top-level principle: **system-optimization** (the *objective function* — what to maximize). The existing standards are all optimization standards. **Ethics/integrity is categorically different: a *constraint* that bounds the space all optimization runs inside, not another axis to maximize.** A peer standard can be traded off; a floor cannot.

Proposed: add **Integrity** as a co-equal top-level entry — different in kind from system-optimization.

- **System-optimization** = the objective function (what to maximize).
- **Integrity** = the constraint (what must never be violated while maximizing).

Floor components (these bind *always*): **honesty by default · non-maleficence (avoid foreseeable harm) · transparency about uncertainty and limitations · accountability (own and make attributable your actions).** This names the latent parent of the spec's existing hard-rules ("never claim a side-effect succeeded without evidence," "no secrets / fail closed," output-hygiene) — they are already honesty and non-maleficence operating as floors; this makes the structure legible.

*Foundation: Beauchamp & Childress (*Principles of Biomedical Ethics*); professional codes (ACM, IEEE); common-morality honesty norms.*

## C. Sharpenings to existing standards (no new entries)

- **#0 System-optimization** — add the *delay/lag* nuance to second-order effects (consequences ripple and lag; delays cause oscillation/overshoot). Foundation: Meadows, Forrester, Senge.
- **#0 System-optimization** — add **purpose-from-behavior (POSIWID)**: a system's real purpose is revealed by what it does, not what it claims. Pairs with the honesty floor as a truth-test. Foundation: Beer, Meadows.
- **Output-hygiene / Audience-tailoring** — add **cognitive-load minimization** (Sweller) as the underlying rationale (minimize extraneous load), rather than a new standard.
- **Conformance / standards application** — add a one-line **expertise-reversal** caveat (Kalyuga & Sweller): scaffolding optimal for a novice operator can be friction for an expert one, so standards should permit graceful escalation/skipping.

## D. Proposed new domain section — "Knowledge & Learning"

Domain-specific practices that are valuable when the task *is* learning or building a knowledge base, but would bloat the general standards:

- Note-craft: progressive summarization, evergreen/atomic notes, the CODE pipeline (Capture-Organize-Distill-Express), linking over foldering. *(Sources: Forte, Ahrens, Matuschak, Luhmann.)*
- Learning mechanics (Bjork "desirable difficulties" cluster): spacing, interleaving, retrieval practice (testing effect); deliberate practice. *(Sources: R. & E. Bjork; Ericsson; Roediger & McDaniel, *Make It Stick*.)*

**Flagged tension to state explicitly:** *linking-over-foldering* appears to push against **#12 hierarchy-first / one-home-per-fact**. Reconciliation: keep one canonical home per fact (hierarchy) and *cross-reference* rather than duplicate — which is what hierarchy-first already prescribes. Worth a sentence so the two aren't read as contradictory.

## E. Situational ethics — domain section, not general

- **Informed consent** (act on a person's behalf only with their understanding/agreement) and **conflicts of interest** (disclose and manage divided loyalties) trigger only under specific conditions (human/stakeholder agency), so they belong in a domain section, not the always-on floor. *(Sources: Beauchamp & Childress; Belmont Report; professional codes.)*

## I. Sharpenings to existing standards (no new entries)

- **#0 System-optimization / #13 Streamlined maintenance** — name the **PDCA improvement loop**: log wins/misses/surprises; **convert repeated fixes into standards**; run a lightweight periodic review to prune/merge/retire. This is the *mechanism* by which this whole standards practice grows (it also describes the domain-examination method's own cadence). Foundation: Deming/Shewhart PDCA; TPS kaizen.
- **#12 Hierarchy-first** — add the explicit **locality-of-override** phrasing observed in the source ("when a narrower/more-specific rule contradicts the general one, the narrower wins"): broad defaults, narrow overrides, and the most-specific applicable rule controls. Sharpens, doesn't replace.
- **#1 Specifications** — reinforce with the SDD framing that **the spec is the source of truth and drift is the failure mode** ("without a robust spec layer, agents drift"); already core to the standard, worth a one-line nod.

## J. Proposed new domain section — "AI agent operations"

Domain-specific practices, valuable when *building or operating AI agents/automations* but too narrow for the general standards:

- **Harness as product:** an agent-context index file (e.g. `AGENTS.md`) as a short *index, not a manual*, pointing to structured/versioned context; linter-/check-enforced invariants; doc-gardening to prune stale context. *(OpenAI harness engineering.)*
- **Context-loading discipline:** prefer lazy/on-demand references (navigated when needed) over eager-loading everything into context; use explicit links (not plain-text names) for context an agent *must* load; batch independent reads. *(Phil's Playbook/Learnings.)*
- **Trigger & recurrence design:** high-signal triggers; recurrence scoped (e.g. skip non-work days early). *(Playbook.)*
- **Mutation safety pre-flight:** view before editing; never run destructive whole-content replaces against structural/secret/unknown blocks; anchor targeted edits to adjacent stable text; when in doubt, ask. *(Learnings — a domain instance of the spec's reliability/fail-closed + always-gated rules.)*
